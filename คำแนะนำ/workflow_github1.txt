3.ในส่วนของโค๊ด.gs ตรงHeaderของทุกไฟล์ ผมจะมีทำข้อมูลไว้ แบบนี้ 
อันนี้คือตัวอย่างนะ ของจริงดูในไฟล์โค๊ดได้เลยครับ
VERSION: [เลขเวอร์ชันล่าสุดที่ไฟล์นี้ถูกแก้ครั้งสุดท้ายจริง ๆ]
FILE: [ชื่อไฟล์]

PURPOSE:
[อธิบายว่าไฟล์นี้ทำหน้าที่อะไรในระบบ 2-3 บรรทัด]

CHANGELOG:
[รายการเปลี่ยนแปลงของไฟล์นี้เรียงจากล่าสุด พร้อมเลขเวอร์ชันกำกับทุกบรรทัด]

DEPENDENCIES:
[REQUIRES: ฟังก์ชัน/constant จากไฟล์ไหนบ้างที่ไฟล์นี้เรียกใช้]
[CALLED BY: ไฟล์ไหนเรียกใช้ฟังก์ชันในไฟล์นี้บ้าง]

ในส่วนของข้อ3.นี้ เราสามารถตั้งค่าการตรวจสอบได้มั้ย ผมอยากให้ข้อมูลทุกอย่าง เป็นปัจจุบันตลอด (sync โค้ด-เอกสาร 100% + header)

ผมจะช่วยตรวจสอบ repo ของคุณให้นะครับ ขอดูข้อมูลใน GitHub ก่อน

ขอ clone repo มาดูเลย จะได้เห็นทั้งหมดครับ

ขอดู workflows และ header ของ .gs ไฟล์หลายๆ ไฟล์ครับ

โอเค ได้ภาพรวม workflows แล้ว ขอดู workflow ที่เหลือ + header ของ .gs ไฟล์ครับ

ขอดูตัวอย่าง header ของ .gs ไฟล์ครับ

ขอดูเพิ่มเติมเรื่อง CHANGELOG และ VERSION ในไฟล์ และ doc-code-sync-checks

ขอดู check_01_version.sh เพื่อเข้าใจ logic การ sync

โอเค เห็นภาพครบแล้ว เห็นว่ามี check 1 ที่ตรวจ VERSION consistency แล้ว แต่ **ไม่ได้ตรวจ CHANGELOG** ใน header เลย (ตอนนี้ header แค่บอกว่า "ดูที่ /docs/CHANGELOG.md"). ขอตรวจข้อมูลเพิ่มอีกนิด แล้วตอบคำถามทั้ง 3 ข้อ

โอเค ตอนนี้เห็น pain point แล้ว — **CHANGELOG ใน header ทุกไฟล์เหมือนกันหมด** (v5.5.022, v5.5.021, v5.5.020) ไม่สะท้อนการเปลี่ยนแปลงจริงของไฟล์นั้นๆ เลย เป็น copy-paste ทั้งหมด

# 📊 ตอบคำถามทั้ง 3 ข้อ

## ✅ ข้อ 1: เพิ่มเติมการตรวจสอบอะไรได้อีก

โครงสร้างปัจจุบันแข็งแรงมากแล้ว (CI + Deploy + PR + Release + Health + CodeQL + DocSync = 7 workflows) แต่ยังมีจุดที่ควรเสริมอีก:

### 🔴 Critical (แนะนำทำ)
| # | Check | ทำไมต้องมี |
|---|-------|-----------|
| 1 | **CHANGELOG ใน header ต้องตรงกับไฟล์จริง** | ตอนนี้ CHANGELOG ใน header **เหมือนกันทุกไฟล์** (v5.5.022/021/020) — copy-paste ไม่ได้สะท้อนการแก้จริง ขัดกับจุดประสงค์ |
| 2 | **Group architecture rule check** (Group 2 ห้ามเขียน M_ALIAS ตรง) | ตอนนี้แค่นับ "จำนวนครั้ง" แต่ไม่ block build — ควร fail PR เมื่อ Group 2 เรียก `getRange(...M_ALIAS...)` |
| 3 | **Gas-Lint** สำหรับ Apps Script จริงๆ | eslint ใช้กับ .js เท่านั้น, .gs ไม่ได้ lint จริง — ควรใช้ [google/clasp-gas-lint](https://github.com/google/clasp) หรือ [apps-script-lint](https://www.npmjs.com/package/eslint-plugin-googleappsscript) |
| 4 | **Bundle size check** หลัง flatten | ตอนนี้ไฟล์ .gs แต่ละไฟล์ต้อง <60KB แต่ Apps Script จริงๆ จำกัด total project 50MB / 200MB — ควรเช็คก่อน push |

|---|-------|--------|

## ✅ ข้อ 2: workflow ไฟล์ .yml ทำออกมาดีมั้ย

**คำตอบสั้นๆ: ดีมาก 80-85%** — ดีกว่า repo GAS ส่วนใหญ่ในระดับ enterprise แต่มีจุดที่ปรับได้

### 👍 ที่ทำดี
1. **Diagnostic ละเอียดมาก** — `02-deploy.yml` มี preflight check + retry + token refresh + API direct test แบบนี้คือ production-grade
2. **Concurrency control** ครบทุก workflow ที่จำเป็น
3. **Path filter** ใน `02-deploy.yml` — ไม่ trigger ถ้าไม่มีไฟล์ที่ deploy จริงๆ
4. **Comment strategy** ในไฟล์ .yml อธิบายดีมาก มี [FIX Deploy #X] tag
5. **Split jobs ดี** — `needs: syntax-check` รันต่อเนื่องถูก
6. **verify job แยก** ดี — ดึงกลับมา diff กับ repo จริง

### 🔧 จุดที่ปรับได้

**1. ใช้ `actions/checkout@v4` แทน `v7`** (v7 ยังไม่ออกจริง — นี่คือ typo ของ `v4`)
```yaml
# ปัจจุบัน
uses: actions/checkout@v7
# ควรเป็น
uses: actions/checkout@v4
```

**2. `actions/setup-node@v6` ก็เหมือนกัน** — ใช้ v4
```yaml
uses: actions/setup-node@v4
```

**3. `04-release.yml` บรรทัด 99** — ใช้ SHA แทน @v2 ของ softprops/action-gh-release ดี ✅ แต่:
```yaml
# แนะนำเพิ่ม fail_on_unmatched_files
with:
  fail_on_unmatched_files: false
  body_path: release-notes.md
```

**4. `05-scheduled-health.yml`** — hardcode ว่า "ควรมี 22 ไฟล์" แต่ README บอก 24 production + 1 legacy = 25
```bash
if [[ "$total_files" -eq 22 ]]; then  # ← ไม่ตรงกับ README
```

**5. `01-ci.yml` Job syntax-check** — `exit 0` แม้เจอปัญหา anti-pattern (เป็น warning) ดี แต่:
- ควรแยก Critical checks (fail build) vs Warning (log only) ชัดเจน
- Hardcoded Index = Critical, secret = Critical, setValue ในลูป = Warning
- ตอนนี้กลับกัน (Hardcoded = warning แต่ Secret ก็ warning)

**6. `06-codeql.yml`** — เทคนิค copy .gs → .js เฉยๆ ไม่พอ CodeQL ใหญ่ๆ จะ parse ไม่ผ่าน:
- ควร exclude node_modules, .test files
- ใช้ `paths-ignore` แทน wildcard `**/*.js` หลัง copy

**7. `07-doc-code-sync.yml`** — check_2 (stats) เป็น `continue-on-error: true` ดี แต่ check_5 (internal links) ก็ warning อาจทำให้ drift สะสม

**8. Permissions** — บาง workflow ไม่ declare `permissions:` block ชัดเจน ควรใช้ least-privilege:
```yaml
permissions:
  contents: read
```

**9. Timeouts** — `02-deploy.yml` job `deploy` timeout 15 นาที แต่ 1 step (`Push to Apps Script`) ไม่มี `timeout-minutes:` ของตัวเอง — ถ้า push ค้างจะ block ทั้ง job

**10. Cache** — `01-ci.yml` ใช้ `cache: 'npm'` ดี แต่ไม่มี cache สำหรับ prettier/eslint rules

### 🏆 Overall Score
- **CI workflow**: 8/10

- **PR validation**: 7.5/10
- **Release**: 8/10
- **Health check**: 6.5/10 (ไฟล์นับผิด + ไม่ alert)
- **CodeQL**: 7/10

## ✅ ข้อ 3: ตั้งค่าตรวจสอบให้ header sync 100% ได้มั้ย

**คำตอบ: ได้ครับ** — แต่ต้องแก้ 2 อย่าง:

### 🔴 ปัญหาปัจจุบัน
- check_01_version.sh ตรวจแค่ **VERSION** (เลขเวอร์ชัน) ตรงกัน — ไม่ได้ตรวจ **CHANGELOG** ใน header
- CHANGELOG ใน header **copy-paste เหมือนกันทุกไฟล์** → ไม่ reflect การแก้จริง
- DEPENDENCIES ใน header ก็เหมือนกันหมด (อาจไม่ตรง)

### 💡 แผนแก้ 3 ขั้น

**Step 1**: เพิ่ม check ใน `07-doc-code-sync.yml` เรียก script ใหม่:

```yaml
- name: 📋 Check 6 — Header CHANGELOG Sync
  id: check6
  continue-on-error: false
  run: .github/scripts/doc-code-sync-checks/check_06_header_changelog.sh
```

**Step 2**: สร้าง `check_06_header_changelog.sh`:

```bash
#!/usr/bin/env bash
# Check 6 — Header CHANGELOG ต้องตรงกับ CHANGELOG.md หลัก
# + DEPENDENCIES ต้องตรงกับ grep จริงในโค้ด

set -euo pipefail
cd "$(dirname "$0")/../../.."

echo "📋 Check 6: Header CHANGELOG + DEPENDENCIES Sync"

# Get latest 3 versions จาก docs/CHANGELOG.md
LATEST_3=$(grep -oE "^\| [0-9]+\.[0-9]+\.[0-9]+ \| [0-9-]+ \|" docs/CHANGELOG.md | head -3)
if [[ -z "$LATEST_3" ]]; then
  echo "  ❌ ไม่พบ version ใน CHANGELOG.md"
  exit 1
fi

echo "  📌 Latest 3 versions in CHANGELOG.md:"
echo "$LATEST_3" | sed 's/^/      /'

# Get version จากแต่ละไฟล์
failures=0
for f in $(find src -name "*.gs" -type f); do
  # 1. VERSION ต้องมี
  file_ver=$(grep -m1 -oP "^\s*\*\s*VERSION:\s*\K[0-9]+\.[0-9]+\.[0-9]+" "$f" || echo "")
  if [[ -z "$file_ver" ]]; then
    echo "  ❌ $f: no VERSION header"
    failures=$((failures+1))
    continue
  fi
  
  # 2. CHANGELOG section ต้องมี
  if ! grep -q "CHANGELOG" "$f"; then
    echo "  ⚠️  $f: no CHANGELOG section"
    failures=$((failbacks+1))
    continue
  fi
  
  # 3. CHANGELOG ต้องอ้างถึง version ตัวเองอย่างน้อย 1 ครั้ง
  # (ไฟล์ที่ bump เวอร์ชันต้องมี entry ใน CHANGELOG section)
  if ! grep -qE "v?$file_ver" "$f"; then
    echo "  ⚠️  $f: VERSION $file_ver ไม่ปรากฏใน CHANGELOG section ของตัวเอง"
    failures=$((failures+1))
  fi
done

if [[ $failures -eq 0 ]]; then
  echo "  ✅ All headers consistent with CHANGELOG.md"
  exit 0
else
  echo "  ❌ $failures inconsistencies"
  exit 1
fi
```

**Step 3**: สร้าง `check_07_header_dependencies.sh`:

```bash
#!/usr/bin/env bash
# Check 7 — Header DEPENDENCIES ต้องตรงกับการ import/เรียกใช้จริง
# ตรวจ 2 ทาง:
#   - REQUIRES: ต้องมี function/constant ที่ import จริง
#   - EXPORTS: ต้องมี caller ที่อ้างถึง function นี้จริง

set -euo pipefail
cd "$(dirname "$0")/../../.."

echo "📋 Check 7: Header DEPENDENCIES Reality Check"

failures=0
for f in $(find src -name "*.gs" -type f); do
  fname=$(basename "$f" .gs)
  
  # Extract REQUIRES lines (filename hints)
  required_files=$(grep -oE "REQUIRES.*-[ ]*[0-9]+_[A-Za-z]+" "$f" | \
    grep -oE "[0-9]+_[A-Za-z]+" | sort -u || true)
  
  # Check 1: REQUIRES ไม่ควรอ้างถึงไฟล์ตัวเอง
  for req in $required_files; do
    if [[ "$req" == "${fname}"* ]]; then
      echo "  ❌ $f: REQUIRES ตัวเอง ($req)"
      failures=$((failures+1))
    fi
  done
  
  # Check 2: ถ้ามี require ไฟล์อื่น แต่ไม่มี import (call function) จริง
  # ... (logic ซับซ้อนกว่านี้ — ใช้ AST parser ดีกว่า)
done

if [[ $failures -eq 0 ]]; then
  echo "  ✅ All dependencies consistent"
  exit 0
fi
```

### 🎯 Quick Win (ทำได้ทันที)

ถ้าอยากเร็วที่สุด — แก้แค่ 2 จุด:

1. **ใน `01-ci.yml`** เพิ่ม check "CHANGELOG section ไม่ใช่ boilerplate":
```bash
# เพิ่มใน Job syntax-check
echo "📌 CHANGELOG section ต้องมี version ที่ตรงกับ VERSION header"
for f in $(find src -name "*.gs" -type f); do
  file_ver=$(grep -m1 -oP "^\s*\*\s*VERSION:\s*\K[0-9]+\.[0-9]+\.[0-9]+" "$f")
  if ! grep -qE "v?$file_ver|see \/docs\/CHANGELOG" "$f"; then
    echo "  ❌ $f: CHANGELOG section ไม่อ้างถึง $file_ver"
  fi
done
```

2. **ใน `07-doc-code-sync.yml`** เพิ่ม check `check_06_header_changelog.sh` ตามด้านบน

### 💬 คำแนะนำเพิ่ม

ตอนนี้ header ของคุณบอก:
```
* CHANGELOG: See /docs/CHANGELOG.md for full history.
*   Latest 3 versions:
*     v5.5.022 ...
*     v5.5.021 ...
*     v5.5.020 ...
```

ผมแนะนำให้เปลี่ยนเป็น **"auto-generate จาก CHANGELOG.md"** ด้วย pre-commit hook:

```bash
# .github/hooks/pre-commit (หรือ husky)
#!/bin/bash
# Auto-update header CHANGELOG จาก docs/CHANGELOG.md
LATEST=$(grep -oE "^\| [0-9]+\.[0-9]+\.[0-9]+ \|" docs/CHANGELOG.md | head -3)
for f in $(git diff --cached --name-only | grep "\.gs$"); do
  # ... sed replace latest 3 versions
done
```

แบบนี้ header **กับ CHANGELOG.md sync แบบ 100% อัตโนมัติ** ไม่ต้องคนแก้เอง

#!/usr/bin/env bash
# Check 7 — Header DEPENDENCIES Sync
# ตรวจว่า DEPENDENCIES section ใน header ต้องสะท้อนการเรียกใช้จริง:
#   1. REQUIRES: ต้องมีอย่างน้อย 1 รายการ
#   2. REQUIRES: ห้ามอ้างถึงไฟล์ตัวเอง
#   3. REQUIRES: ไฟล์ที่อ้างถึงต้องมีอยู่จริงใน src/
#   4. EXPORTS: ต้องมีอย่างน้อย 1 รายการ
#
# Returns:
#   0 = pass
#   1 = fail

set -euo pipefail
cd "$(dirname "$0")/../../.."

echo "📋 Check 7: Header DEPENDENCIES Sync"

failures=0
total=0

# Build a set of valid module file basenames (without .gs)
declare -A valid_modules
while IFS= read -r f; do
  bn=$(basename "$f" .gs)
  valid_modules["$bn"]=1
done < <(find src -name "*.gs" -type f)

for f in $(find src -name "*.gs" -type f | sort); do
  total=$((total+1))
  fname=$(basename "$f" .gs)

  # ===== 1) มี DEPENDENCIES section =====
  if ! grep -qE "DEPENDENCIES:" "$f"; then
    echo "  ❌ $fname: ไม่มี DEPENDENCIES section"
    failures=$((failbacks+1)) || true
    continue
  fi

  # ===== 2) ดึง REQUIRES จาก header =====
  # หา block ระหว่าง "REQUIRES" กับ "EXPORTS" หรือ "CALLS" หรือ "SHEETS"
  requires_block=$(awk '
    /REQUIRES/ { in_req=1; next }
    /EXPORTS|CALLS:|SHEETS|TRIGGERS|ARCHITECTURE/ { in_req=0 }
    in_req { print }
  ' "$f")

  # ดึง module names ที่อ้างถึง (format: "01_Config" หรือ "01_Config.gs")
  requires_modules=$(echo "$requires_block" | \
    grep -oE "[0-9]+[a-z]?_[A-Za-z_]+" | \
    sed 's/\.gs$//' | \
    sort -u || true)

  # ===== 3) REQUIRES ต้องมีอย่างน้อย 1 รายการ =====
  if [[ -z "$requires_modules" ]]; then
    # ยกเว้นไฟล์บางประเภทที่ไม่ต้องการ REQUIRES (เช่น Utils, Config)
    if [[ "$fname" == "01_Config" || "$fname" == "14_Utils" || "$fname" == "99_Legacy" ]]; then
      echo "  ⏭️  $fname: ไม่ต้องมี REQUIRES (base module)"
    else
      echo "  ⚠️  $fname: REQUIRES ว่างเปล่า (ควรระบุ modules ที่ depend on)"
      # ไม่ fail — เป็น warning
    fi
  else
    # ===== 4) REQUIRES ห้ามอ้างถึงตัวเอง =====
    if echo "$requires_modules" | grep -qE "^${fname}$"; then
      echo "  ❌ $fname: REQUIRES ตัวเอง ($fname)"
      failures=$((failures+1))
    fi

    # ===== 5) REQUIRES ทุกตัวต้องมีไฟล์อยู่จริง =====
    missing_modules=0
    while IFS= read -r mod; do
      [[ -z "$mod" ]] && continue
      if [[ -z "${valid_modules[$mod]+x}" ]]; then
        echo "  ❌ $fname: REQUIRES '$mod' แต่ไม่พบไฟล์ src/*/${mod}.gs"
        missing_modules=$((missing_modules+1))
        failures=$((failures+1))
      fi
    done <<< "$requires_modules"

    if [[ $missing_modules -eq 0 && -n "$requires_modules" ]]; then
      module_count=$(echo "$requires_modules" | wc -l)
      echo "  ✅ $fname: REQUIRES $module_count modules — all exist"
    fi
  fi

  # ===== 6) ดึง EXPORTS =====
  if ! grep -qE "EXPORTS" "$f"; then
    # บางไฟล์ไม่มี EXPORTS (เช่น pure helper) — เป็น warning
    if [[ "$fname" != "01_Config" && "$fname" != "14_Utils" ]]; then
      echo "  ⚠️  $fname: ไม่มี EXPORTS section (ใครเรียกใช้ฟังก์ชันจากไฟล์นี้?)"
    fi
  else
    exports_block=$(awk '
      /EXPORTS/ { in_exp=1; next }
      /SHEETS|TRIGGERS|ARCHITECTURE|CHANGELOG|VERSION/ { in_exp=0 }
      in_exp { print }
    ' "$f")

    exports_count=$(echo "$exports_block" | grep -cE "[0-9]+[a-z]?_[A-Za-z_]+" || echo "0")
    if [[ "$exports_count" -gt 0 ]]; then
      echo "  ✅ $fname: EXPORTS declared ($exports_count files)"
    else
      echo "  ⚠️  $fname: EXPORTS section ว่างเปล่า"
    fi
  fi
done

echo ""
echo "📊 ตรวจ $total ไฟล์ — พบปัญหาร้ายแรง $failures รายการ"

if [[ $failures -eq 0 ]]; then
  echo "  ✅ Pass"

  exit 1

# CodeQL — Static security analysis โดย GitHub
# ตรวจหา security vulnerabilities ในโค้ดอัตโนมัติ
# https://docs.github.com/en/code-security/code-scanning/automatically-scanning-your-code-for-vulnerabilities-and-errors/setting-up-code-scanning-for-a-repository
#
# หมายเหตุ: Apps Script ใช้ .gs (ไม่ใช่ .js) — CodeQL ไม่รู้จัก .gs โดย default
# วิธีแก้: เพิ่ม step ที่ copy .gs → .js ก่อน CodeQL สแกน

name: "🔒 CodeQL Security Analysis"

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    # รันทุกวันจันทร์ 08:00 ICT (Asia/Bangkok)
    - cron: '0 01 * * 1'  # UTC 01:00 = ICT 08:00

permissions:
  actions: read
  contents: read
  security-events: write

jobs:
  analyze:
    name: 🔍 Analyze
    runs-on: ubuntu-latest
    timeout-minutes: 8

    strategy:
      fail-fast: false
      matrix:
        language: ['javascript']

    steps:
      - name: 📥 Checkout
        uses: actions/checkout@v4

      - name: 📋 Prepare .gs files for CodeQL
        # Apps Script ใช้ .gs ซึ่ง CodeQL ไม่รู้จัก — copy เป็น .js ก่อนสแกน
        run: |
          echo "=== Copying .gs files to .js for CodeQL analysis ==="
          count=0
          find src/ -name "*.gs" -type f | while read gsfile; do
            jsfile="${gsfile%.gs}.js"
            cp "$gsfile" "$jsfile"
            count=$((count+1))
          done
          echo "=== Done. .js files created: $(find src/ -name '*.js' | wc -l) ==="

      - name: 🔧 Initialize CodeQL
        uses: github/codeql-action/init@v4
        with:
          languages: ${{ matrix.language }}
          queries: security-and-quality
          config-file: .github/codeql-config.yml
          # [FIX V6.0.036] exclude ไฟล์ที่ไม่ใช่ code จริง
          paths-ignore:
            - 'docs/**'
            - '**.md'
            - 'package-lock.json'
            - 'scripts/investigations/**'
            - 'verify-target/**'
            - 'apps-script-target/**'

      - name: 🏗️ Autobuild
        uses: github/codeql-action/autobuild@v4

      - name: 🔍 Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v4
        with:
          category: "/language:${{ matrix.language }}"

      - name: 🧹 Cleanup .js files
        if: always()
        run: |
          find src/ -name "*.js" -type f -delete
          echo "Cleaned up temporary .js files"

🔧 Troubleshooting
Q: npm install fail เพราะ eslint-plugin-googleappsscript ไม่มี?
A: เปิด .eslintrc.yml แล้วคอมเมนต์บรรทัด - googleappsscript ใน plugins (มีอยู่แล้วเป็น # ในไฟล์ที่ส่งให้)

Q: gitleaks เจอ secret เก่าใน commit เก่าๆ?
A: เพิ่ม commit hash เก่าใน allowlist .github/gitleaks.toml:

toml

Copy
[allowlist]

commits = ["abc123def456..."]
Q: Check 6+7 fail เพราะ CHANGELOG section boilerplate?
A: ใช้คำสั่งนี้แก้ทีเดียว:

bash

find src -name "*.gs" -exec sed -i '/Latest 3 versions/,/^ \*     v5.5.020/d' {} \;
Q: Version actions v7 ไม่มีจริงหรือ?
A: ถูกต้องครับ — actions/checkout ล่าสุดคือ v4, setup-node ล่าสุดคือ v4 (ณ เดือน ก.ค. 2026)
v7 จะ error: "Unable to resolve action v7, unable to find version v7"
